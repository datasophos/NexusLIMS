1r"""The NexusLIMS back-end software. 

2 

3This module contains the software required to monitor a database for sessions 

4logged by users on electron microscopy instruments. Based off this information, 

5records representing individual experiments are automatically generated and 

6uploaded to the front-end NexusLIMS CDCS instance for users to browse, query, 

7and edit. 

8 

9Example 

10------- 

11In most cases, the only code that needs to be run directly is initiating the 

12record builder to look for new sessions, which can be done by running the 

13:py:mod:`~nexusLIMS.builder.record_builder` module directly: 

14 

15```bash 

16$ python -m nexusLIMS.builder.record_builder 

17``` 

18 

19Refer to :ref:`record-building` for more details. 

20 

21**Configuration variables** 

22 

23The following variables should be defined as environment variables in your 

24session, or in the ``.env`` file in the root of this package's repository. 

25See the ``.env.example`` file for more documentation and examples. 

26 

27(NexusLIMS-file-strategy)= 

28 

29`NX_FILE_STRATEGY` 

30 Defines the strategy used to find files associated with experimental records. 

31 A value of ``exclusive`` will `only` add files for which NexusLIMS knows how 

32 to generate preview images and extract metadata. A value of ``inclusive`` 

33 will include all files found, even if preview generation/detailed metadata 

34 extraction is not possible. 

35 

36(NexusLIMS-ignore-patterns)= 

37 

38`NX_IGNORE_PATTERNS` 

39 The patterns defined in this variable (which should be provided as a 

40 JSON-formatted string) will be ignored when finding files. A default value 

41 is provided in the ``.env.example`` file that should work for most users, 

42 but this setting allows for further customization of the file-finding routine. 

43 

44(nexuslims-user)= 

45 

46`NX_CDCS_USER` 

47 The username used to authenticate to CDCS API 

48 

49(nexuslims-pass)= 

50 

51`NX_CDCS_PASS` 

52 The password used to authenticate to CDCS API 

53 

54(nexuslims-cdcs-url)= 

55 

56`NX_CDCS_URL` 

57 The root URL of the NexusLIMS CDCS front-end. This will be the target for 

58 record uploads that are authenticated using the CDCS credentials. 

59 

60(nexuslims-instrument-data-path)= 

61 

62`NX_INSTRUMENT_DATA_PATH` 

63 The path (should be already mounted) to the root folder containing data 

64 from the Electron Microscopy Nexus. This folder is accessible read-only, 

65 and it is where data is written to by instruments in the Electron 

66 Microscopy Nexus. The file paths for specific instruments (specified in 

67 the NexusLIMS database) are relative to this root. 

68 

69(nexuslims-data-path)= 

70 

71`NX_DATA_PATH` 

72 The root path used by NexusLIMS for various needs. This folder is used to 

73 store the NexusLIMS database, generated records, individual file metadata 

74 dumps and preview images, and anything else that is needed by the back-end 

75 system. 

76 

77(nexuslims-db-path)= 

78 

79`NX_DB_PATH` 

80 The direct path to the NexusLIMS SQLite database file that contains 

81 information about the instruments in the Nexus Facility, as well as logs 

82 for the sessions created by users using the Session Logger Application. 

83 

84(nexuslims-log-path)= 

85 

86`NX_LOG_PATH` 

87 Directory for application logs. If not specified, defaults to 

88 ``${NX_DATA_PATH}/logs/``. Logs are organized by date: ``logs/YYYY/MM/DD/`` 

89 

90(nexuslims-records-path)= 

91 

92`NX_RECORDS_PATH` 

93 Directory for generated XML records. If not specified, defaults to 

94 ``${NX_DATA_PATH}/records/``. Successfully uploaded records are moved to 

95 an 'uploaded' subdirectory upon upload. 

96 

97(nexuslims-cert-bundle-file)= 

98 

99`NX_CERT_BUNDLE_FILE` 

100 If needed, a custom SSL certificate CA bundle can be used to verify 

101 requests to the CDCS or NEMO APIs. Provide this value with the full 

102 path to a certificate bundle and any certificates provided in the 

103 bundle will be appended to the existing system for all requests made 

104 by NexusLIMS. 

105 

106(nexuslims-cert-bundle)= 

107 

108`NX_CERT_BUNDLE` 

109 As an alternative to ``NX_CERT_BUNDLE_FILE``, you can provide the entire 

110 certificate bundle as a single string (this can be useful for CI/CD 

111 pipelines). Lines should be separated by a single ``\n`` character. If 

112 defined, this value will take precedence over ``NX_CERT_BUNDLE_FILE``. 

113 

114(nexuslims-file-delay-days)= 

115 

116`NX_FILE_DELAY_DAYS` 

117 Controls the maximum delay between observing a session ending and when 

118 the files are expected to be present. For the number of days set (can be 

119 a fraction of a day, if desired), record building will not fail if no 

120 files are found, and the builder will search again until the delay has 

121 passed. For example, if the value is ``2``, and a session ended Monday 

122 at 5PM, the record builder will continue to try looking for files until 

123 Wednesday at 5PM. Default is ``2.0`` days. 

124 

125(nexuslims-local-profiles-path)= 

126 

127`NX_LOCAL_PROFILES_PATH` 

128 Directory for site-specific instrument profiles. These profiles customize 

129 metadata extraction for instruments unique to your deployment without 

130 modifying the core NexusLIMS codebase. Profile files should be Python 

131 modules that register ``InstrumentProfile`` objects. If not specified, 

132 only built-in profiles will be loaded. 

133 

134(nemo-address)= 

135 

136`NX_NEMO_ADDRESS_X` 

137 The path to a NEMO instance's API endpoint. Should be something like 

138 ``https://www.nemo.com/api/`` (make sure to include the trailing slash). 

139 The value ``_X`` can be replaced with any value (such as 

140 ``NX_NEMO_ADDRESS_1``). NexusLIMS supports having multiple NEMO reservation 

141 systems enabled at once (useful if your instruments are split over a few 

142 different management systems). To enable this behavior, create multiple 

143 pairs of environment variables for each instance, where the suffix ``_X`` 

144 changes for each pair (`e.g.` you could have ``NX_NEMO_ADDRESS_1`` paired with 

145 ``NX_NEMO_TOKEN_1``, ``NX_NEMO_ADDRESS_2`` paired with ``NX_NEMO_TOKEN_2``, etc.). 

146 

147(nemo-token)= 

148 

149`NX_NEMO_TOKEN_X` 

150 An API authentication token from the corresponding NEMO installation 

151 (specified in ``NX_NEMO_ADDRESS_X``) that 

152 will be used to authorize requests to the NEMO API. This token can be 

153 obtained by visiting the "Detailed Administration" page in the NEMO 

154 instance, and then creating a new token under the "Tokens" menu. Note that 

155 this token will authenticate as a particular user, so you may wish to set 

156 up a "dummy" or "functional" user account in the NEMO instance for these 

157 operations. 

158 

159(nemo-strftime-fmt)= 

160(nemo-strptime-fmt)= 

161 

162`NX_NEMO_STRFTIME_FMT_X` and `NX_NEMO_STRPTIME_FMT_X` 

163 These options are optional, and control how dates/times are sent to 

164 (`strftime`) and interpreted from (`strptime`) the API. If "`strftime_fmt`" 

165 and/or "`strptime_fmt`" are not provided, the standard ISO 8601 format 

166 for datetime representation will be used (which should work with the 

167 default NEMO settings). These options are configurable to allow for 

168 support of non-default date format settings on a NEMO server. The formats 

169 should be provided using the standard datetime library syntax for 

170 encoding date and time information (see :ref:`strftime-strptime-behavior` 

171 for details). 

172 

173(nemo-tz)= 

174 

175`NX_NEMO_TZ_X` 

176 Also optional; If the "`tz`" option is provided, the datetime 

177 strings received from the NEMO API will be coerced into the given timezone. 

178 The timezone should be specified using the IANA "tz database" name (see 

179 https://en.wikipedia.org/wiki/List_of_tz_database_time_zones). This option 

180 should not be supplied for NEMO servers that return time zone information in 

181 their API response, since it will override the timezone of the returned 

182 data. It is mostly useful for servers that return reservation/usage event 

183 times without any timezone information. Providing it helps properly map 

184 file creation times to usage event times. 

185 

186(email-smtp-host)= 

187 

188`NX_EMAIL_SMTP_HOST` 

189 SMTP server hostname for sending email notifications from the 

190 ``nexuslims-process-records`` script (e.g., ``smtp.gmail.com``). 

191 Required for email notifications. 

192 

193(email-smtp-port)= 

194 

195`NX_EMAIL_SMTP_PORT` 

196 SMTP server port. Default is ``587`` for STARTTLS. 

197 

198(email-smtp-username)= 

199 

200`NX_EMAIL_SMTP_USERNAME` 

201 SMTP username for authentication (if required by your SMTP server). 

202 

203(email-smtp-password)= 

204 

205`NX_EMAIL_SMTP_PASSWORD` 

206 SMTP password for authentication (if required by your SMTP server). 

207 

208(email-use-tls)= 

209 

210`NX_EMAIL_USE_TLS` 

211 Use TLS encryption for SMTP connection. Default is ``true``. 

212 

213(email-sender)= 

214 

215`NX_EMAIL_SENDER` 

216 Email address to send notifications from. Required for email notifications. 

217 

218(email-recipients)= 

219 

220`NX_EMAIL_RECIPIENTS` 

221 Comma-separated list of recipient email addresses for error notifications 

222 (e.g., ``admin@example.com,team@example.com``). Required for email notifications.
